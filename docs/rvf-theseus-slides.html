Project Theseus · RVF 设计演化与定位

RVF：Theseus 的第一个真实承载对象

Project Theseus 是一个「众筹数据 → 迭代 AI 能力」的平台； Review-Validate-Fix 是它的第一个真实承载对象（target）。
RVF 不是单纯的代码审查工具，而是一个把 agent 工作过程结构化的 workflow。
核心问题：AI 完成工作后，如何被观察、审查、修复与交接？

RVF 的价值不在于"再跑一个 reviewer"，而在于把 agent 工作从聊天结果变成一个可治理对象——这正是它能作为 Theseus target 的前提。
01 · 起点

起点：Post-work Quality Loop

  • 最初目标：AI 完成工作后，自动进入 review → validate → fix → handoff
  • 审查不依赖主 agent 的自觉，也不依赖聊天记忆
  • 每次工作都留下可复盘的范围、证据与交接材料
↺ 未通过则回到 Validate 迭代
Review → Validate → Fix → Handoff交接下一棒
Review-Validate-Fix × Project Theseus
这是 RVF 最早的设计核心：为 AI 工作补上一个明确的质量闭环。
02 · 形态演化

从 Skill 到 Workflow Plugin

  • 从单个 skill 演化为跨 Codex / Claude Code / manual path 的 workflow plugin
  • agent-facing instruction 只描述入口与边界
  • 真正状态由 scripts、artifacts、ledger、scope contract 持有

03 · 范围

范围成为一等对象

  • 不再默认把 git diff 当作审查真相
  • 每次 review 都冻结 scope，形成 scope contract / review packet
  • 为多 session、多 worktree、多 agent 协作提供基础
git diff不再默认 = 审查真相 冻结 → Scope Contract / Review Packet每次 review 冻结的范围
↑ 共享同一冻结范围，明确"谁负责哪段工作"
多 session 多 worktree 多 agent
Review-Validate-Fix × Project Theseus
Theseus 要管理的不只是任务，还包括"谁对哪一段工作负责"。RVF 先把这个问题显式化。

  • reviewer 输出、validate/fix 结果、handoff 都成为可审查 artifact
  • "完成 / 已审查 / 已修复 / 可交接"被拆成不同状态

05 · 系统定位

RVF：Theseus 的第一个真实 target

  • Project Theseus：众筹数据 → 迭代 AI 能力 的平台
  • RVF 是它的第一个真实 target——因为 agent 工作已被结构化为可治理对象
  • 关系是双向的：RVF 既是能力评估的信号来源，也是能力落地的试验场

06 · 控制面

控制面：当前过渡 → 未来独立

  • 当前用思迹 / Cline Kanban 作过渡控制面：task、worktree、checkpoint、diff viewer、follow-up
  • 把 review / fix 从主会话分离出来，让 workflow 先真实跑起来
  • 未来走向独立 Kanban：管理 task、agent session、review scope、handoff、checkpoint、人类决策
  • 这条控制面演化与 Project Theseus 是两条独立的线
